# 재개발 실행 체크리스트 v1

이 문서는 `redevelopment-priority-table.md`를 실제 착수 가능한 단위로 분해한 체크리스트다.  
지금 단계에서는 **구현보다 기준 고정과 검증 설계**를 먼저 완료하는 것이 목적이다.

## P0 — 지금 바로 고정해야 하는 항목

### P0-1. 운영 시나리오 기준선 고정
- [ ] 사용자 로그인 흐름 기준 확정
- [ ] 사용자 홈 화면의 핵심 의미 확정
- [ ] 관리자 메인 화면의 우선 정보 확정
- [ ] 예외자 처리 목록 확정
- [ ] “왜 안 되는지” 문구 원칙 확정
- 산출물: `operations-scenario-baseline.md`
- 선행조건: 없음

### P0-2. 근무표 판정 / 예외자 처리 기준 고정
- [ ] 오늘 근무 대상자 판정 기준 문서화
- [ ] 늦조/비근무/연차/반차/예비군 처리 기준 정리
- [ ] Google Sheet 입력 포맷별 판정 차이 정리
- [ ] 이름 중복 가정의 현실성 확인
- 산출물: 판정 기준표, 예외 처리 비교표
- 선행조건: P0-1

### P0-3. 출결 저장 무결성 설계
- [ ] 동일 사용자 동일 액션 중복 저장 방지 방식 정리
- [ ] 기록 순서 보장 방식 정리
- [ ] 동시 요청 시 저장 원자성 요구사항 정리
- [ ] 관리자 정정과 일반 기록 충돌 시 처리 원칙 정리
- 산출물: 무결성 설계 메모, 동시성 요구사항 목록
- 선행조건: P0-1

### P0-4. 공유 검증 규칙 기준 정리
- [ ] 시간 판정 규칙의 단일 기준 정의
- [ ] 위치 판정 규칙의 단일 기준 정의
- [ ] 순서 판정 규칙의 단일 기준 정의
- [ ] UI/API/lib 어디까지가 표시 역할이고 어디부터가 진짜 판정인지 명시
- 산출물: 공유 검증 규칙 표
- 선행조건: P0-1, P0-2

## P1 — 운영 안정화와 경험 보존

### P1-1. 관리자 운영 콘솔 재정렬
- [ ] overview에서 가장 먼저 보여야 할 정보 정리
- [ ] 관리자 작업 흐름(조회/정정/동기화) 우선순위 정리
- [ ] 설정 중심 정보와 운영 중심 정보 분리
- 산출물: 관리자 화면 우선순위 메모
- 선행조건: P0-1

### P1-2. 기록 정정 / 감사로그 흐름 강화

- [ ] before/after/audit log 필수 항목 정리
- [ ] 운영자가 이해하기 쉬운 정정 흐름 문서화
- 산출물: 정정 시나리오 문서, 감사로그 확인 항목
- 선행조건: P0-3

### P1-3. Google Sheet 연동 안정화
- [ ] legacy / monthly / simple 포맷별 처리 차이 정리
- [ ] 파싱 실패 시 운영 대응 방식 정리
- [ ] 시트는 읽기 전용이라는 원칙 재확인
- 산출물: 시트 연동 운영 메모

### P1-4. 사용자 화면 단순성 유지
- [ ] 버튼 수와 화면 정보량 기준 정리
- [ ] 비활성 사유 문구 톤 정리
- [ ] 모바일 우선 인터랙션 기준 정리
- 산출물: 사용자 화면 UX 원칙 메모
- 선행조건: P0-1

## P2 — 범위 정리와 품질 체계

### P2-1. 점심 / 세부 TBM 이벤트 범위 확정
- [ ] 현재 확장된 이벤트 목록 정리
- [ ] 원본 목적 대비 유지/축소 판단
- [ ] phase 1 범위와 실제 구현 차이 문서화
- 산출물: scope decision 메모
- 선행조건: P0-1

### P2-2. 자동 검증 체계 정리
- [ ] smoke 테스트 대상 흐름 목록 정리

- [ ] 동시성 검증이 필요한 시나리오 목록 정리
- 산출물: 검증 매트릭스
- 선행조건: P0-3, P0-4

### P2-3. 배포 / 운영 준비 체크
- [ ] 환경변수 목록 정리
- [ ] 운영 점검 절차 정리
- [ ] 배포 후 확인 항목 정리
- 산출물: 운영 준비 체크리스트
- 선행조건: P1-1, P2-2

## 실행 순서 요약

1. 먼저 **무엇을 유지해야 하는지**를 문서로 고정한다.
2. 그다음 **무결성과 공유 검증 규칙**을 설계한다.
3. 그 후에 관리자/사용자 경험을 정리한다.
4. 마지막으로 범위 확정, 자동 검증, 배포 준비로 넘어간다.

## 왜 지금 문서화를 먼저 하는가

- 현재 가장 큰 리스크는 “기능이 없는 것”보다 “잘못된 기준으로 계속 구현하는 것”이다.
- 원본 GAS 앱은 이미 목적 적합성이 높았기 때문에, 성급한 구현은 오히려 원본 장점을 흐릴 수 있다.
- 특히 이 프로젝트는 병목 예방이 핵심이라, 기준 없이 구현을 늘리면 나중에 다시 저장 구조와 검증 구조를 뜯어고칠 가능성이 크다.
- 그래서 지금 문서화는 일을 미루는 것이 아니라, **잘못된 구현을 줄이는 선행 작업**이다.
