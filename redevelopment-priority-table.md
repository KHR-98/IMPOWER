# 재개발 우선순위표 v1

이 문서는 `app-development-plan.md`의 재개발 전략을 실제 실행 순서로 풀어쓴 우선순위표다.  
원칙은 **기존 GAS 앱의 목적성과 운영 감각을 유지하면서**, **저장·검증·동시성 구조를 약 100명 규모 사용에 맞게 재설계**하는 것이다.

| 우선순위 | 영역 | GAS에서 반드시 보존할 것 | 재개발에서 바꿀 것 | 왜 지금 중요한가 | TDD / QA 증거 |
|---|---|---|---|---|---|
| P0 | 운영 시나리오 기준선 | 사용자와 관리자가 느끼는 흐름, 현재 시간대 의미, 예외자 처리 감각 | 문서 부재 상태를 해소하고 기준 시나리오를 고정 | 기준선 없이 구현을 계속하면 원본 목적성이 흐려짐 | 기능 매핑표, 시나리오 체크리스트, 리뷰 기록 |
| P0 | 근무표 판정 / 예외자 처리 | 오늘 근무 대상자 판정, 늦조·비근무·연차·반차·예비군 처리 | 시트 파싱과 내부 모델을 서버 중심으로 일관화 | 잘못된 근무 판정은 전체 운영 신뢰도를 바로 해침 | 샘플 시트 케이스 검증, 판정 결과 비교표 |
| P0 | 출결 저장 무결성 | 출근/TBM/퇴근 기록의 의미와 순서 | 중복 요청 방지, idempotency, unique 제약, 원자적 저장 | 이번 재개발의 핵심 명분이 병목 예방이기 때문 | 중복 클릭 테스트, 동일 사용자 동시 요청 테스트 |
| P0 | 공유 검증 규칙 | 시간/위치/순서 판정의 일관성 | UI/route/lib 간 중복 규칙 제거, 서버 판정 단일화 | 규칙이 갈라지면 운영 결과가 화면마다 달라짐 | route/API/화면 동일 결과 검증 |
| P1 | 관리자 운영 콘솔 | 지금 봐야 할 사람과 단계가 먼저 보이는 운영 감각 | 설정 중심 화면보다 운영 중심 정렬 강화 | 관리자가 즉시 판단할 수 있어야 원본 장점이 유지됨 | 관리자 시나리오 점검, overview 검토 체크리스트 |
| P1 | 기록 정정과 감사로그 | 예외 상황을 관리자만 정정, 사유와 이력 보존 | 정정 플로우를 더 안전하고 추적 가능하게 다듬음 | 운영상 예외는 반드시 생기고, 추적 가능해야 함 | 정정 전후 비교, audit log 확인 |
| P1 | Google Sheet 연동 | 근무표를 읽기 전용 원본으로 사용하는 방식 | 시트 입력 포맷 처리 안정화, 앱 DB 중심 운영 | 원본과의 연결고리를 유지하면서도 병목은 줄여야 함. 단, 근무 판정 정확도에 직접 영향을 주는 파싱 신뢰성은 P0 근무표 판정 항목과 함께 본다. | legacy/monthly/simple 포맷 검증 |
| P1 | 사용자 화면 단순성 | 버튼이 적고, 왜 안 되는지 바로 이해되는 UX | 내부는 복잡해져도 화면은 더 단순하게 유지 | 제품 목적성은 기능 수보다 즉시성에 있음 | 모바일 QA, 비활성 사유 확인 |
| P2 | 점심/세부 TBM 이벤트 | 원본 운영상 필요한 세부 단계 의미 | 현재 확장된 이벤트를 유지할지 단순화할지 확정 | 현재 구현이 계획보다 앞서 있어 scope drift 위험이 있음 | 기능 유지/축소 결정 기록 |
| P2 | 자동 검증 체계 | 실제 운영 흐름이 깨지지 않는 확인 | Playwright 안정화, 회귀 smoke 확대 | 병목 대응은 수동 확인만으로는 부족함 | e2e smoke, typecheck, build |
| P2 | 배포/운영 준비 | 현장 운영자가 계속 쓸 수 있는 안정성 | 환경변수, 배포 절차, 장애 대응 절차 정리 | 구현 후 운영 전환에서 가장 많이 흔들리는 영역 | 배포 체크리스트, 운영 매뉴얼 |

## 실행 메모

- P0는 “지금 당장 기준을 고정하지 않으면 이후 구현이 흔들리는 항목”이다.
- P1은 “운영 현실에서 자주 쓰이는 흐름을 안정화하는 항목”이다.
- P2는 “핵심 목적은 아니지만 품질과 지속 가능성을 높이는 항목”이다.
- 각 항목 구현 전에는 `무엇을 보존할지`와 `무엇을 바꿀지`를 먼저 분리해서 판단한다.
