# 운영 시나리오 기준선 v1

이 문서는 재개발 과정에서 **무엇을 반드시 보존해야 하는지**를 고정하는 기준 문서다.  
핵심 원칙은 **기존 GAS 앱의 목적성과 운영 감각을 유지하면서**, 저장·검증·동시성 구조만 더 안정적으로 재설계하는 것이다.

## 1. 이 문서의 목적

- 구현 전에 사용자/관리자 관점의 기준 흐름을 고정한다.
- 이후 변경이 “기능 확장”인지 “원본 경험 훼손”인지 판단할 수 있게 한다.
- 예외 처리와 운영 판단 기준을 구현보다 먼저 합의한다.

## 2. 사용자 기준 시나리오

### 2-1. 로그인

- 사용자는 이름 기반 ID와 비밀번호로 로그인한다.
- 로그인 실패 시 최소한 아래는 구분되어야 한다.
  - 계정 없음
  - 비밀번호 오류
  - 비활성 계정
- 로그인 이후에는 역할에 따라 `/dashboard` 또는 `/admin`으로 이동한다.

### 2-2. 사용자 홈 화면

- 사용자는 화면에 들어오자마자 아래를 이해할 수 있어야 한다.
  - 오늘 날짜
  - 본인 이름
  - 오늘 근무 대상 여부
  - 현재 시간대 의미
  - 지금 할 수 있는 행동
- 버튼은 기본적으로 **출근 / TBM / 퇴근**의 운영 의미를 중심으로 보인다.
- 현재 구현에 점심/세부 TBM 이벤트가 있더라도, 이후 유지 여부는 원본 목적 기준으로 다시 판단한다.

### 2-3. 출근

- 출근은 아래 조건을 모두 만족할 때만 허용한다.
  - 오늘 근무 대상자
  - 출근 가능 시간
  - 출입 zone 내부
  - 아직 출근 전
- 조건이 맞지 않으면 버튼을 숨기기보다, **왜 안 되는지 이해 가능한 문구**를 보여준다.

### 2-4. TBM

- TBM은 아래 조건을 모두 만족할 때만 허용한다.
  - 오늘 근무 대상자
  - TBM 가능 시간
  - TBM zone 내부
  - 선행 조건 충족
- 사용자는 “왜 지금 TBM이 안 되는지”를 즉시 이해할 수 있어야 한다.

### 2-5. 퇴근

- 퇴근은 아래 조건을 모두 만족할 때만 허용한다.
  - 오늘 근무 대상자
  - 퇴근 가능 시간
  - 출입 zone 내부
  - 선행 조건 충족
- 퇴근 기록은 출근/TBM과 동일하게 서버가 최종 판정한다.

## 3. 관리자 기준 시나리오

### 3-1. 관리자 메인 화면

- 관리자는 설정 화면보다 먼저 **오늘 누가 어디까지 했는지**를 봐야 한다.
- 화면은 아래 질문에 즉시 답해야 한다.
  - 오늘 근무 대상은 몇 명인가
  - 출근 미완료자는 누구인가
  - TBM 미완료자는 누구인가
  - 퇴근 미완료자는 누구인가
  - 관리자 정정이 발생했는가

### 3-2. 근무표 동기화

- 관리자에게는 오늘 근무표를 다시 읽어오는 수단이 있어야 한다.
- 근무표는 Google Sheet를 **읽기 전용 원본**으로 사용한다.
- 동기화는 운영 판단을 돕기 위한 것이지, 출결 결과를 Sheet에 다시 쓰기 위한 것이 아니다.

### 3-3. 기록 정정

- 예외 상황의 최종 해결자는 관리자다.
- 다만 정정은 반드시 아래를 만족해야 한다.
  - 관리자 권한
  - 정정 사유 입력
  - 변경 이력 보존
- 정정은 편의 기능이 아니라 **감사 가능한 예외 처리 절차**여야 한다.

### 3-4. 설정 관리

- zone과 GPS 기준은 운영자가 조정할 수 있어야 한다.
- 시간 설정은 현재 구현상 제한적으로 열려 있으므로, 확대 여부는 원본 운영 목적과 실제 운영 필요를 먼저 확인한다.

## 4. 예외 처리 기준

다음 예외는 재개발에서 반드시 운영 의미를 보존해야 한다.

- 비근무자
- 늦조
- 연차
- 반차
- 예비군
- GPS 부정확
- zone 외부
- 허용 시간 외
- 선행 기록 미완료

원칙은 단순하다.

- 예외는 숨기지 않는다.
- 사용자는 **왜 안 되는지** 알아야 한다.
- 관리자는 **무엇이 막혔는지** 한눈에 봐야 한다.

## 5. 서버 판정 기준

- 위치 판정은 서버가 한다.
- 시간 판정은 서버가 한다.
- 기록 순서 판정은 서버가 한다.
- 중복 저장 방지는 서버/DB가 책임진다.
- 클라이언트는 편의 UI를 제공하되, 최종 진실은 서버가 가진다.

## 6. 재개발에서 반드시 유지할 제품 감각

- 기능이 많아 보여도 안 된다.
- 사용자는 지금 할 행동만 명확히 보여야 한다.
- 관리자는 지금 밀린 단계를 빠르게 알아야 한다.
- 설정 구조보다 운영 흐름이 더 앞에 와야 한다.
- 내부 구조가 복잡해져도 화면 경험은 단순해야 한다.

## 7. 이 문서를 어떻게 사용할지

- 새 기능을 넣기 전에 이 문서 기준으로 **보존할 것 / 바꿀 것**을 먼저 나눈다.
- 구현 결과가 맞는지 볼 때는 “기능이 동작하나”만 보지 말고, “원본 운영 감각이 유지되나”를 같이 본다.
- 이 문서는 `app-development-plan.md`, `redevelopment-priority-table.md`, `AGENTS.md`와 함께 재개발의 기준선으로 사용한다.
