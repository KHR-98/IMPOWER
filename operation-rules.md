# 출퇴근통합시스템 운영 규칙 확정안

작성일: 2026-03-19
기준 입력: 사용자 운영 규칙 응답
상태: 구현 기준 문서

## 1. 범위 정리
이번 입력 기준으로 기존 1차 범위보다 아래 항목이 추가되었다.
- 주간조 / 늦조 분리
- TBM 다단계 처리
  - 오전 TBM
  - 오후 TBM
  - 퇴근 TBM
- 점심 입문 / 출문 기록
- 월별 시트 탭 구조를 읽는 Google Sheet 파서

즉, 현재 앱의 단순 `출근 / TBM / 퇴근` 3단계 구조만으로는 실제 운영 규칙을 충분히 표현할 수 없다.
다음 구현은 이벤트 기반 구조로 확장하는 방향을 기본 전제로 한다.

## 2. 운영 목표
- 가장 중요한 목적
  - 출근 정확도 향상
  - TBM 인원 파악 자동화
  - 관리자 집계 자동화
- 우선순위
  - 정확성 > 운영 편의성 > 속도 > 디자인
- 1차 오픈 기준
  - 관리자와 대원들이 실제로 매일 출근, TBM, 퇴근 기록을 사용할 수 있으면 오픈

## 3. 근무표 운영 규칙
- 근무표 원본은 Google Sheet다.
- 근무표 관리자는 본인(부팀장)이다.
- 문서 이름은 `메모리 PCS 근무표`다.
- 실제 시트 탭 이름은 월별로 바뀐다.
  - 예: `2월`, `3월`
- 시트 구조는 일반적인 테이블이 아니라 월간 표 형태다.
  - B열: 날짜
  - C열: 요일
  - D~K열: 이름
  - L열: 비고
  - N, O열: 사무 인원
- 날짜 형식은 `YYYY-MM-DD` 기준으로 앱에 맞춰 정규화한다.

### 근무 대상자 판정
- 시트에 별도로 작성된 인원은 늦조 또는 주말 근무 인원으로 본다.
- 전체 인원 중 늦조와 비고 대상자를 제외한 인원을 주간조 근무 대상자로 본다.
- 비고에는 연차, 반차, 예비군 등 근무 제외 사유가 들어간다.
- 비근무자인데 긴급 투입된 경우에는 시트를 수정한 뒤 수동 동기화한다.

### 동기화 규칙
- 기본 동기화는 하루 1회 자동 동기화를 목표로 한다.
- 자동 동기화 기준 시각은 새벽 또는 출근 전 준비 시간대인 오전 5시 전후를 기본값으로 둔다.
- 관리자는 필요할 때 수동 동기화를 추가로 실행할 수 있다.
- 근무표 예외 처리는 별도 기능보다 시트 수정 또는 관리자 정정으로 처리한다.

### 이름 매칭 규칙
- 현재 가정: 이름 중복 없음
- 1차 기본 매칭 기준
  - 앱의 `display_name`과 시트 이름을 우선 매칭한다.
- 추후 필요 시 `username` 매칭을 보조 기준으로 둔다.

## 4. 시간 규칙
현재 운영 규칙은 공통 시간대 하나가 아니라 조별 시간대 구조다.

### 주간조
- 출근 가능: `06:00 ~ 08:30`
- 오전 TBM: `06:00 ~ 08:30`
- 점심 입출문: `11:40 ~ 13:30`
- 오전 반차 / 조퇴자 퇴근 처리: `11:40 ~ 13:00`
- 오후 TBM: `13:35 ~ 13:45`
- 퇴근 TBM: `16:30 ~ 16:45`
- 퇴근 가능: `16:30 ~ 18:00`
- 퇴근 현황 확인: `16:50 ~ 18:00`

### 늦조
- 출근 가능: `09:00 ~ 11:00`
- 점심 입출문: `13:50 ~ 15:40`
- 퇴근 가능: `19:30 ~ 21:00`
- 퇴근 현황 확인: `19:50 ~ 21:00`

### 버튼 처리
- 시간 밖 버튼은 비활성화한다.
- 버튼 비활성 이유는 사용자 화면에 표시한다.
- 지각 처리 방식은 추후 결정한다.

## 5. 위치 규칙
- 출근 / 퇴근은 `entry` 지점 안에서만 허용한다.
- TBM은 `tbm` 지점 안에서만 허용한다.
- 예상 출입 지점 수는 4개다.
- TBM 지점 수는 1개로 시작하고 추후 2개까지 늘어날 수 있다.
- 출입 지점 기본 반경은 `100m`를 기본값으로 둔다.
- TBM 지점 기본 반경도 `100m`를 기본값으로 둔다.
- GPS 허용 오차는 `100m`를 기본값으로 둔다.
- GPS 정확도가 허용 오차보다 나쁘면 저장을 차단하고 다시 시도 안내를 보여준다.
- 위치 권한을 거부하면 저장을 차단하고 권한 허용 안내를 보여준다.
- GPS가 잡히지 않거나 버튼이 눌리지 않으면 관리자 정정으로 처리한다.
- 1차 위치 위변조 대응 수준은 완전 차단이 아니라 운영상 억제 수준으로 본다.

### 보충 설명
- 현장 경계가 넓은 경우
  - 하나의 큰 원으로 처리하지 않고 출입구별 zone을 여러 개 둔다.
- 실내 GPS 오차가 심한 경우
  - 반경을 약간 조정하거나 관리자 정정으로 흡수한다.

## 6. 기록 흐름 규칙
현재 운영 흐름은 단일 TBM이 아니라 다단계 이벤트다.

### 기본 이벤트 목록
- 주간조 기본 이벤트
  - 출근
  - 오전 TBM
  - 점심 출문
  - 점심 입문
  - 오후 TBM
  - 퇴근 TBM
  - 퇴근
- 늦조 기본 이벤트
  - 출근
  - 점심 출문
  - 점심 입문
  - 퇴근

### 순서 규칙
- 출근 전 TBM은 불가
- 출근 전 퇴근은 불가
- 같은 항목의 중복 기록은 불가
- 퇴근 후 다시 출근은 1차에서 불가
- 일부 예외 퇴근은 허용한다.
  - 오전 반차
  - 오후 반차
  - 조퇴
  - 몸이 아파서 조기 퇴근
- TBM 없이 퇴근해야 하는 예외는 관리자 정정으로 처리한다.
- 출근만 하고 TBM 미참석한 사람은 관리자 화면에서 별도 확인 가능해야 한다.

## 7. 사용자 계정 규칙
- 로그인 방식은 관리자 생성 ID + 비밀번호다.
- ID는 이름 그대로 사용한다.
- 표시 이름도 실명 그대로 사용한다.
- 이름 중복은 현재 없다고 가정한다.
- 초기 비밀번호는 모두 동일하게 시작하고 나중에 변경한다.
- 비밀번호 최소 길이는 4자다.
- 비활성 계정은 로그인할 수 없다.
- 권한은 `user`, `admin` 두 종류만 사용한다.
- 활성 관리자 계정은 최소 1개 이상 유지해야 한다.

## 8. 관리자 정정 규칙
- 관리자만 정정 가능하다.
- 정정 가능한 항목은 다음과 같다.
  - 출근 시간
  - TBM 시간
  - 퇴근 시간
- 정정 사유는 필수다.
- 정정 이력 저장은 필수다.
- 잘못된 기록 삭제는 시간을 비우는 방식으로 처리할 수 있다.
- 정정 사유는 관리자만 본다.
- 정정 표시 방식은 현재 `이름 옆 (정정)` 또는 별도 배지 중 하나로 처리 가능하다.
  - 현재 앱은 `(정정)` 표시를 우선 사용 중이다.

## 9. 예외 처리 규칙
- GPS가 안 잡히면 관리자 정정
- 현장에 왔는데 버튼이 안 눌리면 관리자 정정
- 근무 대상인데 실제로 쉬는 경우 관리자 정정
- 비근무자인데 긴급 투입된 경우 시트 수정 후 수동 동기화
- TBM 집합에 늦게 합류한 경우 관리자 정정
- 휴대폰 미소지 / 배터리 방전 / 인터넷 불가 상황은 현재 운영상 주요 시나리오로 보지 않는다.

## 10. 관리자 화면 기준
### 상단 수치
- 근무 대상
- 비고 인원
- 시간대별 버튼 체크 완료 인원
- 출근 완료 / 미출근
- 각 TBM 완료 / 미완료
- 퇴근 완료 / 미퇴근

### 사용자별 표 기본 항목 제안
현재 사용자 입력에서 이 항목은 미확정이므로 아래 기준으로 확정 제안한다.
- 이름
- 조 구분(주간조 / 늦조 / 비고)
- 근무 대상 여부
- 출근 시각
- 오전 TBM 시각
- 점심 출문 시각
- 점심 입문 시각
- 오후 TBM 시각
- 퇴근 TBM 시각
- 퇴근 시각
- 정정 여부

### 기타
- 필터 기능은 1차에서 제외
- 정렬은 작성해 둔 이름 순서를 우선
- 다운로드는 엑셀 필요
- 출력 기능은 필요 없음

## 11. 사용자 화면 기준
- 오늘 날짜
- 이름
- 오늘 근무 대상 여부
- 현재 기록 가능한 버튼 목록
- 상태 문구
- 버튼 비활성 이유 표시

### 미확정 항목에 대한 기본 제안
- 현재 위치 상태 표시: 표시함
- 사용자 상태 문구: 이벤트 단계별로 표시
  - 예: 출근 완료, 오전 TBM 대기, 점심 출문 대기, 퇴근 완료
- 관리자 정정 사실 표시: 사용자에게는 숨김, 관리자 화면에서만 표시

## 12. 운영 환경
- 실제 사용자 수: 약 37명
- 베타 이후 100명 수준까지 확장 가능
- 관리자 수: 7명
- 주 사용 기기: 아이폰 / 안드로이드 혼합
- 인터넷 상태: 양호
- 실내 사용 비율: 보통
- 실외 사용 비율: 보통

## 13. 보안 / 접근 기준
- 외부 공개는 하지 않고 사내용만 사용한다.
- 로그인 없이 접근 가능한 화면은 없다.
- 관리자 화면은 관리자 계정만 접근 가능해야 한다.
- 개인정보 민감도는 높지 않지만, 이름 / 위치 / 출퇴근 시간이 포함되므로 운영 데이터로 취급한다.

## 14. 구현상 핵심 결론
현재 구조는 `출근 / TBM / 퇴근` 3단계 고정 모델이다.
하지만 실제 운영 규칙은 아래를 요구한다.
- 조별 시간표
- 다중 TBM
- 점심 입문 / 출문
- 월간 표 형태의 근무표 파싱

따라서 다음 구현 우선순위는 아래 순서가 맞다.
1. 단일 TBM 구조를 이벤트 기반 구조로 리팩터링
2. 주간조 / 늦조 규칙을 이벤트 템플릿으로 분리
3. 관리자 / 사용자 화면을 다중 이벤트 구조로 변경
4. Google Sheet 월간 표 파서 구현
5. 엑셀 다운로드 추가

## 15. 현재 기준 미확정 항목
아래는 구현 전에 한 번 더 확인하면 좋은 항목이다.
- 지각 처리 방식
- 관리자 정정 표시를 사용자 화면에도 보여줄지 여부
- TBM 지점 2개 확장 시 사용자 선택 방식
- 비고 인원 수치 계산 상세 규칙
- 늦조에서 TBM을 전혀 쓰지 않는지 여부
- 점심 입문 / 출문이 모두 위치 검증 대상인지 여부
