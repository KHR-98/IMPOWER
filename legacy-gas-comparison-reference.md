## 2. 기존 Apps Script 서버 코드에서 확인된 실제 구조

### 2.1 앱 기본 구조
기존 시스템은 Google Apps Script 서버가 직접 아래를 처리한다.
- `doGet()`으로 HTML 페이지 반환
- 사용자 인증
- 출근 기록 저장
- TBM 기록 저장
- 퇴근 기록 저장
- 점심 등록 / 출문 / 입문 저장
- 비밀번호 변경
- 관리자 현황판용 집계 데이터 생성

즉, `서버 + 시트 저장 + 관리자 집계`가 Apps Script 한 파일에 집중된 구조다.

### 2.2 시트 구조
확인된 시트는 아래와 같다.
- `전체인원`
  - 3행부터 읽음
  - 이름, 비밀번호, 역할을 시트에서 직접 읽음
- `늦조인원`
  - 날짜별 늦조 인원 목록
  - 날짜별 휴가/비고 인원 목록
  - 일부 예외 인원 판정에도 사용
- `출근기록`
  - 날짜, 이름, 출근시간, 위도, 경도, 주소
- `퇴근기록`
  - 날짜, 이름, 퇴근시간, 위도, 경도, 주소
- `출석기록`
  - 날짜, 이름, 오전 TBM, 오후 TBM, 퇴근 TBM
- `점심`
  - 날짜, 이름, 점심 출문, 점심 입문, 위도, 경도, 주소, 점심 등록

중요한 점은 기존 앱이 `기능별로 시트를 나눠 저장`한다는 것이다.

### 2.3 사용자 인증 방식
기존 서버 코드는 `전체인원` 시트에서 이름과 비밀번호를 직접 읽고 문자열 비교로 인증한다.
즉:
- 비밀번호 해시 없음
- 세션 저장 없음
- 시트가 곧 사용자 DB 역할

현재 앱과 비교하면 이 부분은 구조적으로 가장 큰 차이 중 하나다.

## 3. 기존 Apps Script 서버의 운영 규칙

### 3.1 조 구분 규칙
- `getTodayExempt()`는 사실상 `늦조 인원` 목록을 읽는다.
- `getTodayLeave()`는 휴가/비고 인원을 읽는다.
- `isExceptionUser()`는 `늦조인원` 시트의 특정 추가 컬럼을 보고 점심 모드 예외를 판단한다.

즉, 기존 앱은 `늦조인원` 시트를 중심으로 아래를 함께 판정한다.
- 늦조 여부
- 휴가/비고 여부
- 일부 예외 사용자 여부

### 3.2 시간대(period) 규칙
기존 서버의 핵심은 `getCurrentPeriod()`다.
확인된 period는 아래와 같다.
- `am`: 06:00 ~ 08:30
- `exempt_work`: 09:00 ~ 11:00
- `lunch1`: 11:40 ~ 13:30
- `pm`: 13:35 ~ 13:45
- `lunch2`: 13:50 ~ 15:40
- `off`: 16:30 ~ 16:45
- `workoff`: 16:50 ~ 18:00
- `exempt_workoff`: 19:50 ~ 21:00
- `none`: 나머지 시간

이 구조 때문에 기존 관리자 현황판은 `현재 시간대에 맞는 표만 보여주는 방식`으로 동작한다.

### 3.3 출근 규칙
`markWorkAttendance()` 기준:
- 늦조면 09:00 ~ 11:00만 허용
- 일반 인원이면 06:00 ~ 08:30만 허용
- 회사 기준 좌표 반경 안이어야 함
- 이미 출근 기록이 있으면 차단
- 시트 `출근기록`에 시간과 좌표와 주소 저장

### 3.4 퇴근 규칙
`markWorkOffAttendance()` 기준:
- `isWorkOffTime(name)`이 true여야 함
- 점심 퇴근 11:40 ~ 13:00 허용
- 늦조 퇴근 19:30 ~ 21:00 허용
- 주간조 퇴근 16:30 ~ 18:00 허용
- 중요한 차이:
  - 회사 반경 `안`이면 차단
  - 즉, 기존 서버는 `회사 밖으로 이동한 뒤에만 퇴근 가능`

이 부분은 현재 앱과 규칙이 정반대다.

### 3.5 TBM 규칙
`markAttendance()` 기준:
- TBM 좌표 반경 안이어야 함
- 시간대에 따라 자동으로 컬럼 결정
  - 오전 TBM
  - 오후 TBM
  - 퇴근 TBM
- `출석기록` 시트에 단계별로 저장

즉, 기존 서버는 이미 `다단계 TBM`을 완성해 둔 상태다.

### 3.6 점심 규칙
`recordLunch(name, type, lat, lng)` 기준:
- 허용 시간
  - 점심 1부: 11:40 ~ 13:30
  - 점심 2부: 13:50 ~ 15:40
- 동작 순서
  - `register`: 점심 명단 등록
  - `exit`: 등록된 사람만 점심 출문 가능
  - `entry`: 출문 기록이 있는 사람만 점심 입문 가능
- 저장 위치
  - `점심` 시트

중요한 점:
- 기존 서버는 `점심 등록` 단계가 있다.
- 현재 앱은 아직 `점심 출문 / 입문` 중심이고, `점심 등록` 이벤트는 없다.

### 3.7 관리자 현황 집계 규칙
`getTodayStatus()` 기준:
- 모든 사용자 목록을 기준으로 당일 상태 생성
- 늦조, 휴가, 일반 인원을 함께 계산
- period에 따라 해당 세션 인원만 필터링
- 점심 시간에는 주간조/늦조를 분리해서 보여줌
- 통계 예시
  - 출근 완료
  - 오전 TBM 완료
  - 오후 TBM 완료
  - 퇴근 TBM 완료
  - 퇴근 완료
  - 점심 등록
  - 점심 출문
  - 점심 입문
  - 늦조 대상/면제 대상/휴가 대상

즉, 기존 서버는 관리자용 `현장 운영 대시보드 계산기` 역할을 이미 하고 있다.

## 4. 현재 Next.js 앱 서버 구조 요약

### 4.1 저장 구조
현재 앱은 시트를 나누지 않고 DB 레코드 중심이다.
- `users`
- `roster_entries`
- `attendance_records`
- `audit_logs`
- `zones`
- `app_settings`

현재 방향은 `하루 한 사람 한 레코드` 구조에 이벤트를 확장하는 방식이다.

### 4.2 인증 구조
현재 앱은:
- 사용자 계정을 DB에 저장
- 비밀번호 해시 사용
- 쿠키 세션 사용
- 역할 분리
- 관리자 권한 확인

보안과 구조는 현재 앱이 훨씬 낫다.

### 4.3 검증 구조
현재 앱은:
- 서버에서 시간 검증
- 서버에서 zone 검증
- 서버에서 중복 검증
- 관리자 정정 시 audit log 저장

기존 Apps Script보다 검증 책임이 더 명확하다.

### 4.4 현재 앱의 새 구조
현재 앱은 최근 반영 기준으로:
- 주간조 / 늦조 구분 가능
- 점심 출입 대상자를 따로 지정 가능
- TBM 버튼 하나로 오전/오후/퇴근 TBM 단계 처리 가능
- 점심 출문 / 입문 버튼을 시간대별로 노출 가능

즉, 사용자 화면 구조는 현재 앱이 더 단순하고 확장 가능한 방향이다.

## 5. 핵심 비교 분석

### 5.1 가장 중요한 운영 차이
#### 퇴근 위치 규칙
기존 Apps Script 서버:
- 회사 반경 밖에서만 퇴근 허용

현재 앱:
- 현재 구현은 `entry zone 안에서 퇴근 허용`

이 차이는 단순 구현 차이가 아니라 운영 정책 차이다.
이 부분은 반드시 사용자와 다시 맞춰야 한다.

정리:
- 기존 정책을 따를지
- 현재 정책을 유지할지
를 확정해야 한다.

### 5.2 점심 흐름
기존 Apps Script 서버:
- 점심 등록이 먼저 있어야 함
- 등록한 사람만 출문 가능
- 출문한 사람만 입문 가능

현재 앱:
- 현재는 점심 대상자 지정 후 출문/입문 흐름 중심
- 점심 등록 이벤트는 아직 없음

판단:
- 기존 서버의 점심 로직이 더 엄격하고 운영 친화적이다.
- 현재 앱도 정확히 맞추려면 `점심 등록` 이벤트를 추가할 가치가 높다.

### 5.3 조 구분 데이터 소스
기존 Apps Script 서버:
- `늦조인원` 시트가 조 판정의 기준

현재 앱:
- `roster_entries.shift_type`에 저장
- 관리자 화면에서 당일 늦조/점심 대상 조정 가능
- 장기적으로는 Google Sheet 파서가 이 값을 자동 채워야 함

판단:
- 현재 앱 구조가 더 낫다.
- 다만 기존 서버가 실제로 쓰는 `늦조인원` 시트 구조를 파서로 옮겨야 운영 일관성이 생긴다.

### 5.4 관리자 상태판
기존 Apps Script 서버:
- `currentPeriod` 중심
- 시간대별로 필요한 통계와 표만 보여줌
- 주간조/늦조/점심 세션 분리 뚜렷

현재 앱:
- 요약 카드 + 사용자별 표 중심
- 아직 `period` 기반 표 분기 없음
- 대신 계정관리, 정정, 지점설정, 근무자설정은 더 강함

판단:
- 관리자 현황판 UX는 기존 서버가 더 현장형이다.
- 현재 앱은 운영 도구는 더 좋지만, `지금 당장 누구를 봐야 하는지` 보여주는 감각은 아직 약하다.

### 5.5 데이터 저장 방식
기존 Apps Script 서버:
- 기능별 시트 분산 저장
- 사람이 보기 쉽고 바로 수정 가능
- 하지만 구조 일관성과 확장성이 떨어짐

현재 앱:
- DB 구조화 저장
- API와 검증 로직 분리
- 감사 로그 가능
- 향후 확장 유리

판단:
- 저장 구조는 현재 앱 방향이 맞다.
- 다만 관리자용 출력 형태는 기존 시트/표 스타일을 참고할 가치가 크다.

## 6. 기존 서버 코드에서 꼭 참고해야 할 구현 포인트

### 6.1 `currentPeriod()` 개념
현재 앱에도 거의 그대로 옮길 가치가 있다.
이유:
- 관리자 화면이 훨씬 명확해짐
- 어떤 통계를 보여줄지 결정 가능
- 점심 1부 / 점심 2부 / 늦조 시간대를 자연스럽게 분리 가능

### 6.2 점심 등록 단계
기존 서버는 점심 기능이 아래 3단계라 운영상 강하다.
- 등록
- 출문
- 입문

현재 앱도 장기적으로는 아래 방향이 좋다.
- 관리자 지정만으로 끝내지 말고
- `점심 등록` 이벤트 또는 점심 승인 흐름을 추가

### 6.3 시트 기반 늦조/휴가 판정
기존 서버는 실제 운영 데이터를 `늦조인원` 시트에서 읽는다.
현재 앱도 Google Sheet 연동 시 아래를 구현해야 한다.
- 늦조 자동 판정
- 휴가/비고 자동 판정
- 점심 세션 대상 분리 가능 여부 검토

### 6.4 퇴근 정책 재확인
현재 앱은 `현장 안에서 퇴근`, 기존 서버는 `현장 밖에서 퇴근`이다.
이 차이는 실제 운영 정책 확인이 반드시 필요하다.

## 7. 현재 앱이 더 나은 점
이 부분은 유지하는 것이 맞다.
- 비밀번호 해시
- 세션 기반 인증
- 관리자 정정 및 감사 로그
- DB 저장 구조
- 다중 zone 지원
- 관리자 지점 설정 기능
- 관리자 계정 관리 기능
- 서버 중심 검증

즉, 현재 앱은 기존 서버를 그대로 복제할 대상이 아니라,
기존 서버의 운영 로직을 더 안전한 구조로 다시 구현하는 대상이다.

## 8. 바로 반영 가치가 높은 다음 작업
우선순위 순으로 정리한다.

### 1순위
- `currentPeriod` 서버 유틸 도입
- 관리자 현황판을 period 기반으로 재구성

### 2순위
- 점심 등록 이벤트 추가 여부 확정
- 기존 서버처럼 `등록 -> 출문 -> 입문` 흐름 반영 검토

### 3순위
- Google Sheet 파서를 `늦조인원` 시트 구조에 맞게 설계
- 늦조 / 휴가 / 예외 사용자 판정 자동화

### 4순위
- 퇴근 위치 정책 확정
  - 현장 안 퇴근
  - 현장 밖 퇴근

## 9. 최종 결론
기존 Apps Script 서버는 `실제 운영 규칙`이 이미 많이 녹아 있다.
특히 아래 4개는 현재 앱이 반드시 참고해야 한다.
- `currentPeriod` 개념
- 점심 등록 흐름
- 늦조/휴가 시트 기반 판정
- 퇴근 위치 정책

현재 앱은 구조와 보안은 더 낫지만,
운영 디테일은 아직 기존 서버의 일부만 옮겨온 상태다.

따라서 가장 적절한 방향은 아래다.
- 데이터 저장과 권한 구조는 현재 앱 유지
- 관리자 현황판과 운영 분기는 기존 서버 로직 적극 차용
- 점심 / 늦조 / 예외 처리 규칙은 기존 서버 기준으로 재정의
