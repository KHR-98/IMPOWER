# 예외자 처리 기준표 v1

이 문서는 재개발 과정에서 **예외 상태를 어떻게 해석하고 보여줄지**를 고정하는 기준표다.  
원칙은 기능을 숨기지 않고, **왜 안 되는지 사용자와 관리자가 각각 이해할 수 있게** 만드는 것이다.

## 1. 공통 원칙

- 예외는 서버 판정 기준으로 해석한다.
- 예외가 있으면 버튼만 막지 말고 사유를 보여준다.
- 관리자는 예외 때문에 누가 어느 단계에서 막혔는지 확인할 수 있어야 한다.
- 예외 상황의 최종 처리 수단은 관리자 정정이지만, 정정은 반드시 사유와 이력을 남긴다.

## 2. 예외자 처리 기준표

| 예외 상태 | 운영 의미 | 사용자 화면 원칙 | 관리자 화면 원칙 | 저장/판정 원칙 | 근거 |
|---|---|---|---|---|---|
| 비근무자 | 오늘 근무 대상이 아님 | 출근/TBM/퇴근을 허용하지 않고 비근무 사유를 보여준다 | 오늘 대상자 집계에서 제외되거나 비대상으로 명확히 보여준다 | `isScheduled=false` | `operations-scenario-baseline.md`, `app-development-plan.md`, `lib/types.ts` |
| 늦조 | 오늘 근무 대상이지만 기본 주간조와 다른 시간대 흐름을 따름 | 현재 시간대 의미가 늦조 기준으로 보여야 한다 | 오늘 대상에는 포함되지만 주간조와 구분되어 보여야 한다 | `shiftType='late'` | `operations-scenario-baseline.md`, `lib/types.ts`, `lib/google-sheets.ts` |
| 연차 | 근무표상 당일 근무에서 제외되는 휴무성 예외 | 버튼을 허용하지 않고 근무 제외 의미를 보여준다 | 비대상 또는 예외 처리 대상으로 보이게 한다 | 기본적으로 `isScheduled=false` | `operations-scenario-baseline.md`, `lib/google-sheets.ts` |
| 반차 | 전일 비근무와 동일 취급이 아님. 시간대/행동 제약이 달라질 수 있는 예외 | 단순 비근무로 숨기지 말고, 운영 의미가 확정되기 전까지 별도 검토 대상으로 본다 | 원본 운영 기준을 확인하기 전까지 일반 근무자와 동일 취급하지 않는다 | 현재 파서는 일부 값에서 `scheduled=false`로 해석 가능 | `operations-scenario-baseline.md`, `app-development-plan.md`, `lib/google-sheets.ts` |
| 오전반차 / 오후반차 | 반차의 세부 유형 | 사용자에게는 가능한 행동이 달라질 수 있음을 전제로 다뤄야 한다 | 관리자에게는 단순 반차보다 구체적으로 확인 가능해야 한다 | 현재 Google Sheet 이름 토큰에서는 예외/제외성 값으로 취급됨 | `lib/google-sheets.ts` |
| 예비군 | 출결 일반 흐름에서 벗어나는 공인 예외 | 기본 출결 버튼을 일반 근무자와 동일하게 열지 않는다 | 관리자에게는 예외자로 분명히 보여야 한다 | 현재 이름 토큰 기준 예외/제외성 값으로 취급됨 | `operations-scenario-baseline.md`, `lib/google-sheets.ts` |
| GPS 부정확 | 위치는 읽혔지만 신뢰도가 낮음 | 저장하지 않고 재시도 안내를 보여준다 | 사용자 문제인지 현장 문제인지 확인 가능해야 한다 | `accuracy > maxGpsAccuracyM` 이면 저장 차단 | `app-development-plan.md`, `operations-scenario-baseline.md` |
| zone 외부 | 위치는 읽혔지만 허용 구역 밖임 | 버튼 비활성 + 위치 사유를 보여준다 | 누가 위치 조건 때문에 막혔는지 보이게 한다 | 서버가 반경 판정 후 저장 차단 | `app-development-plan.md`, `operations-scenario-baseline.md` |
| 허용 시간 외 | 위치/근무 대상과 무관하게 지금은 허용 시간대가 아님 | 지금 왜 안 되는지 시간 사유를 보여준다 | 현재 시간대 기준 미완료자 판단에 반영한다 | 서버 시간대 판정으로 차단 | `app-development-plan.md`, `operations-scenario-baseline.md`, `lib/types.ts` |
| 선행 기록 미완료 | 기록 순서가 맞지 않음 | 예: 출근 전 TBM, 출근 전 퇴근은 막는다 | 관리자도 어느 선행 단계가 비었는지 이해해야 한다 | 서버가 순서 판정 후 차단 | `app-development-plan.md`, `operations-scenario-baseline.md` |
| 이름 매칭 실패 | 시트 이름이 앱 사용자와 연결되지 않음 | 사용자 화면 이전에 운영상 해결해야 하는 문제 | 관리자에게 `unmatched` 상태로 보여야 한다 | 출결 대상자로 자동 연결하지 않는다 | `lib/google-sheets.ts`, `lib/types.ts` |

## 3. 지금 기준으로 확정된 것과 미확정인 것

### 확정된 것
- 비근무자는 일반 출결 흐름에 들어가지 않는다.
- 늦조는 별도 shift로 구분한다.
- GPS 부정확, zone 외부, 허용 시간 외, 선행 기록 미완료는 서버가 차단한다.
- 관리자 정정은 예외 상황의 최종 해결 절차다.

### 아직 운영 기준 재확인이 필요한 것
- 반차를 완전 비근무처럼 볼지, 일부 시간대 허용 상태로 볼지
- 오전반차 / 오후반차를 현재 세부 이벤트 구조와 어떻게 연결할지
- 예비군을 단순 비근무로 볼지 별도 집계 대상으로 둘지

## 4. 문서 사용 방법

- 구현 전에는 이 표를 기준으로 예외를 신규 추가할지, 기존 예외를 세분화할지 판단한다.
- 구현 후에는 “버튼이 막혔다”가 아니라 “왜 막혔는지 설명 가능하냐”로 검증한다.
- 미확정 항목은 실제 구현에 바로 넣지 말고 운영 기준부터 다시 확인한다.
